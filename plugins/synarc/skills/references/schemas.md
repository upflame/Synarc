# Purpose

Canonical document schemas for the cognition layer brain files. Every `.md` file follows the universal pattern: frontmatter (YAML), H1 title, H2 sections, compact bullets. No prose padding.

# Scope

The 9 brain document schemas (B1-B9). Does not cover file system structure, CI artifacts, or external tool formats.

# Notes

All brain files live under `/brain/` in the repository root. Snapshots go in `/brain/snapshots/`. The CHANGELOG_INTELLIGENCE lives at `/brain/CHANGELOG_INTELLIGENCE.md`.

## Universal Pattern

Every brain file uses frontmatter + H1 title + H2 sections. Required sections per document type:

| Section | CURRENT_STATE | SYSTEM_MAP | ARCHITECTURE | MODULE_MAP | API_CONTRACTS | FEATURE_LOG | CHANGELOG_INTELLIGENCE | SESSION_LEDGER | SNAPSHOT |
|---|---|---|---|---|---|---|---|---|---|
| # Purpose | ✓ | ✓ | ✓ | ✓ | ✓ | ✓ | ✓ | ✓ | ✓ |
| # Schema sections (per doc below) | ✓ | ✓ | ✓ | ✓ | ✓ | ✓ | ✓ | ✓ | ✓ |

## 1. CURRENT_STATE.md [B1]

**Purpose:** Cognitive memory of the repository at a point in time. Answers: what exists right now?

**Schema:**
- # Active Architecture — modules/services, style, entry points, runtime
- # Major Modules — table: name, responsibility, owner, entry point
- # Important Contracts — REST, events, queues, gRPC; producers and consumers
- # Current Risks — CRITICAL/HIGH/MEDIUM/LOW
- # Extension Paths — safe areas for new functionality
- # Invariants — behaviors and constraints that must always hold
- # Cognitive Summary — one paragraph system overview

## 2. SYSTEM_MAP.md [B2]

**Purpose:** Static-ish structural map of the entire system.

**Schema:**
- # Overview — system name, purpose, scale, deployment model
- # Boundaries — internal vs external, trust boundary
- # Modules — table: name, type (service/lib), responsibility, language/runtime
- # Dependencies — internal (module relationships) and external (DBs, APIs, caches, queues with type, auth, SLA)
- # Data Flow — request/event path through the system
- # External Integrations — table: integration, protocol, purpose, owner, SLA
- # Cognitive Summary

## 3. ARCHITECTURE.md [B3]

**Purpose:** Architectural principles, layers, service design, failure modes.

**Schema:**
- # Principles — list with brief rationale
- # Layers — table: layer, responsibility, modules
- # Services — table: service, type (API/Worker), responsibility, port/queue, replicas
- # Contracts — table: contract, type, producer, consumers, schema location
- # Data Flow — request/event path through layers
- # Failure Modes — table: failure, cause, effect, mitigation
- # Cognitive Summary

## 4. MODULE_MAP.md [B4]

**Purpose:** Index of all modules with responsibilities, owners, entry points.

**Schema:**
- # Module List — table: name, path, type, owner, status
- # Responsibilities — per module: what it does, does not do, exposes, consumes
- # Entry Points — table: module, entry point file, trigger
- # Dependencies — internal/external, hard/soft
- # Ownership — table: module, team, slack channel, on-call
- # Risks — table: module, risk level, description

## 5. API_CONTRACTS.md [B5]

**Purpose:** All API contracts — REST, gRPC, events, queues.

**Schema:**
- # Auth — method, token location, refresh mechanism, scope model
- # Endpoints — per endpoint: method, path, purpose, auth, rate limit, idempotency; request/response schema; status codes; error cases
- # Events / Queue Messages — per event: name, producer, consumers, queue/topic, delivery guarantee; event schema
- # Invariants — what this API always guarantees

## 6. FEATURE_LOG.md [B6]

**Purpose:** Chronological log of features with engineering context.

**Schema (per feature):**
- # Feature: \<Name\> — YYYY-MM-DD
- # Purpose — business + engineering reason
- # Systems Affected — modules/services changed, contracts modified
- # Change Summary — what was added/changed/removed
- # Risks — risk level and description
- # Snapshot Link — → brain/snapshots/...

## 7. CHANGELOG_INTELLIGENCE.md [B7]

**Purpose:** Curated, impact-aware analysis of what changed and why it matters. Not a raw git log.

**Schema (per commit/PR):**
- # Commit: \<SHA or PR\> — YYYY-MM-DD
- # Summary — what changed, at what layer, why
- # Impact — layer changed, breaking?, affected modules, downstream risk
- # Affected Areas — table: file/module, change type, risk
- # Risks — risk level and description
- # Follow-Up — action items, migrations needed, test coverage

## 8. SESSION_LEDGER.md [B8]

**Purpose:** Per-session log of tool calls, risk, and state changes. Generated by session-tracking.

**Schema (per session):**
- # Session: \<id\> — timestamp
- # Scope — declared task, files, risk cap
- # Ledger — table: seq, tool, file, worktype, risk, timestamp
- # Checkpoints — list of checkpoint IDs and timestamps
- # Aggregate Risk — final risk score
- # Files Written — list with rollback info

## 9. /brain/snapshots/\<timestamp\>-\<name\>.md [B9]

**Purpose:** Immutable cognitive snapshot at a point in time. Never updated, only superseded.

**Schema:**
- # Metadata — timestamp, commit SHA, branch, author, PR link
- # Purpose — why this change exists
- # Systems Affected — modules, contracts, config, schema
- # Architecture Changes — structural changes, new boundaries
- # Data Flow — before/after request path
- # Dependencies Added — table: type, name, used by, purpose
- # Extension Points — where future features can be safely added
- # Risks Introduced — table: level, risk, affected module
- # Breaking Changes — what breaks, migration path
- # Cognitive Summary — what changed, why it matters, main risks
